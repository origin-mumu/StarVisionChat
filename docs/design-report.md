# 灵眸星视 设计与实现报告

## 一、产品定位

**灵眸星视** 是一款多模态 AI 视界助理，融合视觉感知、语音交互、场景智能与持久记忆，让用户通过摄像头和麦克风与 AI 进行自然交互。

**核心卖点**：看得见、听得懂、说得出、记得住的 AI 助手。

---

## 二、业务场景

### 场景 1：生活百科 — "这是什么？"

**目标用户**：普通大众，对动植物、美食、物品不熟悉的人

**典型场景**：
- 在公园看到不认识的花，对准它问"这是什么花？"
- 在国外餐厅看到外文菜单，拍下来问"这道菜是什么？"
- 收到快递不认识零件，问"这是什么？怎么用？"

**用户价值**：不用打字描述，指一下就能问

---

### 场景 2：厨房助手 — "帮我看看火候"

**目标用户**：做饭时双手不方便的人

**典型场景**：
- 炒菜时手上沾满油，语音问"帮我看看肉熟了吗？"
- 烘焙时不确定状态，把摄像头对准蛋糕问"这个颜色对吗？"
- 想做一道菜但不知道步骤，边做边问"下一步放什么？"

**用户价值**：双手被占用时，语音 + 摄像头完成交互

---

### 场景 3：学习辅导 — "这道题怎么做？"

**目标用户**：学生、家长辅导孩子

**典型场景**：
- 孩子遇到不会的数学题，对准题目问"这道题怎么解？"
- 看英文文章不认识单词，指着问"这个词什么意思？"

**用户价值**：拍照就能问，不用手动输入题目

---

### 场景 4：老人陪聊 — "你帮我看看这个"

**目标用户**：老年人，打字困难但想用 AI

**典型场景**：
- 老人收到短信不确定是不是诈骗，对准摄像头问"这条消息是真的吗？"
- 想知道药盒说明书，问"这个药怎么吃？"
- 孤独时想聊天，直接语音说话，AI 语音回复

**用户价值**：不需要打字，说话就能用

---

### 场景 5：面试辅导 — "帮我准备一下"

**目标用户**：求职者

**典型场景**：
- 说"我明天有个面试"，AI 自动切换到面试辅导模式
- AI 模拟面试官，进行自我介绍、常见问题练习
- 指着简历问"这段经历怎么描述更好？"

**用户价值**：场景智能识别，自动切换 AI 角色

---

### 场景 6：穿搭建议 — "穿什么好？"

**目标用户**：需要穿搭建议的人

**典型场景**：
- 说"明天约会穿什么"，AI 自动切换到穿搭助手模式
- 把衣服对准摄像头，问"这件搭配好看吗？"
- AI 根据场合、颜色搭配给出建议

**用户价值**：有视觉能力的私人造型师

## 四、成本控制策略

### 4.1 成本来源分析

每次用户交互可能产生的 API 调用：

| 调用类型 | 触发时机 | 成本因素 |
|---------|---------|---------|
| LLM 对话 | 每次发消息 | input + output tokens |
| ASR 语音识别 | 每次语音输入 | audio tokens |
| TTS 语音合成 | 每次语音回复 | tokens |
| 视觉分析 | 每次发消息（含图像） | 图像 token（按分辨率） |
| 场景监控 | 每 2 秒一次 | 图像 token |
| 手势识别 | 合并到场景监控 | 无额外成本 |

### 4.2 控制技巧清单

| 编号 | 技巧 | 描述 | 状态 |
|------|------|------|------|
| C1 | **按需截帧** | 只在用户发消息时捕获帧发送给 LLM | ✅ 已采用 |
| C2 | **多模态合并** | 图像 + 文本一次调用完成，不分开调用 | ✅ 已采用 |
| C3 | **图像压缩** | JPEG 60% 质量，640px 最大宽度 | ✅ 已采用 |
| C4 | **历史裁剪** | 只保留最近 10 轮对话 | ✅ 已采用 |
| C5 | **回复长度限制** | max_tokens=1024 | ✅ 已采用 |
| C6 | **场景监控合并** | 手势识别 + 场景检测一次 API 调用 | ✅ 已采用 |
| C7 | **锁定状态短 prompt** | 锁定时只检测手势，prompt 更短，token 更少 | ✅ 已采用 |
| C8 | **费用追踪** | 实时显示 Token 消耗 | ✅ 已采用 |
| C9 | **本地 STT** | 使用 faster-whisper 免费 | ❌ 未采用（统一 MiMo） |
| C10 | **免费 TTS** | 使用 Edge-TTS 免费 | ❌ 未采用（统一 MiMo） |

### 4.3 成本效果对比

以一次典型对话为例（用户每 10 秒发一次消息，持续 1 分钟）：

| 方案 | API 调用次数 | 说明 |
|------|-------------|------|
| 无控制 | 36 次 | 每 2 秒视觉分析(6次) + 每次对话(6次) = 36 次 |
| 仅按需截帧 | 6 次 | 只有用户发消息时调用 |
| 按需截帧 + 多模态合并 | 6 次 | 每次对话一次多模态调用 |
| **最终方案** | **6 次** | 相比无控制节省 83% |

场景监控（每 2 秒一次）为独立的轻量调用，prompt 简短（max_tokens=10-20），成本可控。

## 五、技术架构

### 5.1 系统架构

```
┌─────────────────────────────────────────────────┐
│  前端 (Vue 3 + Vite + Pinia + vue-router)       │
│  3D 粒子动画 · 摄像头 · 麦克风 · TTS · 主题     │
└──────────────────────┬──────────────────────────┘
                       │ WebSocket (JSON + base64)
┌──────────────────────▼──────────────────────────┐
│  后端 (FastAPI + Python)                         │
│  WebSocket 路由 · 会话管理 · 工具调用 · 记忆      │
└──────────────────────┬──────────────────────────┘
                       │ OpenAI 兼容 API
┌──────────────────────▼──────────────────────────┐
│  AI 服务                                         │
│  MiMo (小米) / Qwen-Omni (阿里)                  │
│  视觉理解 · 语音识别 · 语音合成 · 对话            │
└─────────────────────────────────────────────────┘
```

### 5.2 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 双引擎架构 | MiMo + Qwen-Omni | 互为备份，Qwen Realtime 延迟更低 |
| 按需截帧 | 发消息时捕获 | 节省成本，画面更实时 |
| 场景监控合并 | 手势 + 场景一次调用 | API 调用减半 |
| 会话存储 | localStorage | 无需后端数据库，部署简单 |
| 记忆存储 | SQLite | 轻量持久化，支持 Function Calling |
| 粒子动画 | 原生 Canvas | 零依赖，性能好 |
