# 灵眸星视 设计与实现报告

## 一、产品定位

**灵眸星视** 是一款 AI 视界助理，让用户通过摄像头和麦克风与 AI 进行自然交互。用户可以指着某个东西问"这是什么"，或者边做事情边用语音聊天，AI 能看到画面、听到声音、开口说话。

**核心卖点**：看得见、听得懂、说得出的 AI 助手。

---

## 二、目标用户与业务场景

### 场景 1：生活百科 — "这是什么？"

**目标用户**：普通大众，尤其是对动植物、美食、物品不熟悉的人

**典型场景**：
- 在公园看到一朵不认识的花，打开应用对准它，问"这是什么花？"
- 在国外餐厅看到菜单全是外文，拍下来问"这道菜是什么？辣吗？"
- 收到一个快递，不认识里面的零件，问"这是什么？怎么用？"

**用户价值**：不用打字描述，指一下就能问，降低使用门槛

---

### 场景 2：厨房助手 — "帮我看看火候"

**目标用户**：做饭时双手不方便的人

**典型场景**：
- 炒菜时手上沾满油，用语音问"帮我看看肉熟了吗？"
- 烘焙时不确定状态，把摄像头对准蛋糕问"这个颜色对吗？"
- 想做一道菜但不知道步骤，边做边问"下一步放什么？"

**用户价值**：双手被占用时，用语音 + 摄像头完成交互

---

### 场景 3：学习辅导 — "这道题怎么做？"

**目标用户**：学生、家长辅导孩子

**典型场景**：
- 孩子遇到不会的数学题，把摄像头对准题目，问"这道题怎么解？"
- 看英文文章不认识某个单词，指着问"这个词什么意思？"
- 做实验时不确定步骤，问"下一步应该怎么做？"

**用户价值**：不用手动输入题目，拍照就能问，节省时间

---

### 场景 4：老人陪聊 — "你帮我看看这个"

**目标用户**：老年人，打字困难但想用 AI

**典型场景**：
- 老人收到一条短信不确定是不是诈骗，把手机屏幕对准摄像头问"这条消息是真的吗？"
- 想知道药盒上的说明书写了什么，问"这个药怎么吃？"
- 孤独时想聊天，直接语音说话，AI 语音回复

**用户价值**：不需要打字，说话就能用，降低老年人使用 AI 的门槛

---

### 场景 5：购物比价 — "这个值不值？"

**目标用户**：购物时需要参考意见的人

**典型场景**：
- 在超市看到两款商品，问"这两款有什么区别？哪个更好？"
- 看到一件衣服，问"这个颜色搭配好看吗？"
- 买水果时不确定熟没熟，问"这个西瓜熟了吗？"

**用户价值**：购物时有个"参谋"，快速获得建议

---

### 场景 6：旅行翻译 — "这个写的什么？"

**目标用户**：出国旅行的人

**典型场景**：
- 在国外看到路牌不认识，对准问"这个路牌写的什么？"
- 看到菜单全是外文，问"这个菜单有什么推荐的？"
- 收到酒店的设施说明，问"这个怎么用？"

**用户价值**：拍照就能翻译，不用打字输入

## 三、用户故事：计划 vs 实现

### 计划阶段

在项目初期，我们基于以上业务场景，规划了以下用户故事：

| 编号 | 用户故事 | 对应场景 |
|------|---------|---------|
| US-1 | 用户指着某个物体问 AI "这是什么"，AI 能看到并回答 | 场景 1、3、6 |
| US-2 | 用户用语音跟 AI 对话，AI 用语音回复 | 场景 2、4 |
| US-3 | 用户发消息时，AI 能同时看到当前画面 | 场景 1、2、3、5、6 |
| US-4 | 用户可以管理多个对话，分开不同话题 | 所有场景 |
| US-5 | 用户可以切换界面主题、选择 AI 音色 | 所有场景 |

### 最终实现

| 故事 | 计划方案 | 最终方案 | 变化原因 |
|------|---------|---------|---------|
| US-1 | 每 2 秒自动截帧分析 | 用户发消息时实时捕获帧 | 节省成本，画面更实时 |
| US-2 | Whisper 语音识别 + Edge-TTS | MiMo ASR + MiMo TTS | 统一技术栈，简化配置 |
| US-3 | 视觉分析和对话分开调用 | 多模态一次调用 | 成本减半，架构简化 |
| US-4 | 前端 localStorage 存储 | 按计划实现 | 无变化 |
| US-5 | Element Plus 主题切换 | 6 套主题 + 粒子动画 | 超出预期 |

## 四、成本控制策略

### 4.1 成本来源分析

每次用户交互可能产生的 API 调用：

| 调用类型 | 触发时机 | 成本因素 |
|---------|---------|---------|
| LLM 对话 | 每次发消息 | input + output tokens |
| ASR 语音识别 | 每次语音输入 | audio tokens |
| TTS 语音合成 | 每次语音回复 | tokens |
| 视觉分析 | 每次发消息（含图像） | 图像 token（按分辨率） |

### 4.2 思考过的控制技巧

| 编号 | 技巧 | 描述 | 预期效果 |
|------|------|------|---------|
| C1 | 按需截帧 | 只在用户发消息时捕获帧 | 沉默时零成本 |
| C2 | 多模态合并 | 图像+文本一次调用完成 | 调用次数减半 |
| C3 | 图像压缩 | JPEG 60% 质量，640px 宽度 | 降低图像 token |
| C4 | 历史裁剪 | 只保留最近 10 轮对话 | 控制 input token |
| C5 | 回复长度限制 | max_tokens=1024 | 控制 output token |
| C6 | 本地 STT | 使用 faster-whisper 免费 | 语音识别零成本 |
| C7 | 免费 TTS | 使用 Edge-TTS 免费 | 语音合成零成本 |
| C8 | 费用追踪 | 实时显示 API 调用费用 | 用户主动控制 |

### 4.3 实际采用情况

| 编号 | 技巧 | 采用状态 | 原因 |
|------|------|---------|------|
| C1 | 按需截帧 | ✅ 已采用 | 核心优化，效果显著 |
| C2 | 多模态合并 | ✅ 已采用 | 核心优化，架构简化 |
| C3 | 图像压缩 | ✅ 已采用 | 简单有效 |
| C4 | 历史裁剪 | ✅ 已采用 | 简单有效 |
| C5 | 回复长度限制 | ✅ 已采用 | 简单有效 |
| C6 | 本地 STT | ❌ 未采用 | 用户要求统一 MiMo |
| C7 | 免费 TTS | ❌ 未采用 | 用户要求统一 MiMo |
| C8 | 费用追踪 | ✅ 已采用 | 透明化成本 |

### 4.4 成本效果对比

以一次典型对话为例（用户每 10 秒发一次消息，持续 1 分钟）：

| 方案 | API 调用次数 | 说明 |
|------|-------------|------|
| 无控制 | 36 次 | 每 2 秒视觉分析(6次) + 每次对话(6次) = 36 次 |
| 仅按需截帧 | 6 次 | 只有用户发消息时调用 |
| 按需截帧 + 多模态合并 | 6 次 | 每次对话一次多模态调用 |
| **最终方案** | **6 次** | 相比无控制节省 83% |

## 五、技术架构

### 5.1 系统架构

```
用户设备（浏览器）
    │
    ├── 摄像头 → 实时画面展示 + 发消息时捕获帧
    ├── 麦克风 → 语音录音 + VAD 检测
    └── UI → 会话管理 + 主题切换 + 设置
    │
    ▼
WebSocket 连接
    │
    ▼
后端服务（FastAPI）
    │
    ├── LLM 服务 → MiMo mimo-v2.5 多模态对话
    ├── ASR 服务 → MiMo mimo-v2.5-asr 语音识别
    ├── TTS 服务 → MiMo mimo-v2.5-tts 语音合成
    └── 会话管理 → 对话历史 + 成本追踪
    │
    ▼
MiMo API（TokenPlan）
```

### 5.2 关键技术决策

1. **统一 MiMo 模型**：对话、视觉、ASR、TTS 全部使用 MiMo 系列，简化配置
2. **按需多模态**：用户发消息时实时捕获帧，而非定时分析
3. **会话本地存储**：前端 localStorage 持久化，无需后端数据库
4. **WebSocket 全双工**：实时通信，支持状态推送和音频流传输
